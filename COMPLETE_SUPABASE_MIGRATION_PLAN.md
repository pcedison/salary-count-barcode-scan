# Supabase 遷移完整計劃

## 當前狀態分析

### 本地資料庫內容
- ✅ 5筆薪資記錄（2025年3-7月陳文山）
- ✅ 2位員工資料
- ✅ 1個系統設定
- ✅ 0筆臨時考勤和假日記錄

### 問題識別
1. **連接問題**：Supabase 回報 "Tenant or user not found"
2. **架構問題**：使用 @neondatabase/serverless 而非 postgres 驅動
3. **配置問題**：DATABASE_URL 指向 Neon 而非 Supabase
4. **資料遷移**：5筆關鍵薪資記錄需要完整轉移

## 解決方案：分階段遷移

### 階段1：修復 Supabase 連接
#### 問題根源
- Supabase 專案可能被暫停或密碼已更改
- 需要重新獲取正確的連接字串

#### 解決步驟
1. 重新確認 Supabase 專案狀態
2. 重置資料庫密碼
3. 獲取新的連接字串
4. 測試連接成功

### 階段2：更新系統架構
#### 驅動程式更換
- 移除：@neondatabase/serverless
- 改用：postgres + drizzle-orm/postgres-js

#### 連接層重構
- 統一使用 postgres 驅動
- 移除混合連接邏輯
- 簡化資料庫初始化

### 階段3：資料遷移
#### 匯出現有資料
```sql
-- 匯出薪資記錄
SELECT * FROM salary_records;
-- 匯出員工資料  
SELECT * FROM employees;
-- 匯出系統設定
SELECT * FROM settings;
```

#### 匯入至 Supabase
- 建立完整資料庫結構
- 逐筆匯入歷史薪資記錄
- 驗證資料完整性

### 階段4：系統驗證
- 確認所有 API 正常運作
- 測試薪資計算功能
- 驗證前端顯示正確
- 完整功能測試

## 風險評估與緩解

### 高風險項目
1. **資料遺失**：5筆薪資記錄為客戶關鍵資料
   - 緩解：遷移前完整備份
   - 驗證：遷移後逐筆對比

2. **連接失敗**：Supabase 連接仍無法建立
   - 緩解：準備替代 Supabase 專案
   - 備案：使用 Railway 或 Neon 作為 PostgreSQL 主機

3. **功能中斷**：遷移期間系統無法使用
   - 緩解：非營業時間進行遷移
   - 回滾：保留原系統作為備用
